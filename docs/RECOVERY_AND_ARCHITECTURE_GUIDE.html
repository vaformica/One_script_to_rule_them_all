IDtracker Pipeline Controller

Recovery, architecture, workflow, and handoff guide

This document is intended to preserve the entire design even if the original chat is lost. The repository is the authoritative software record. The scientific source videos and master TOMLs remain on Firebird and are treated as read-only inputs.

Scientific workflow being managed

  1. Videos are recorded and stored on Firebird.
  2. The IDtracker segmentation application is run through VNC on Firebird.
  3. A master TOML is created for each cell or arena.
  4. A video may contain many cells.
  5. Some cells contain one beetle and represent behavioral assays.
  6. Some cells contain two beetles and may include a secondary fungus ROI, representing fight assays.
  7. IDtracker GPU jobs are submitted through SLURM.
  8. Completed sessions are post-processed with Python.
  9. Track PNGs are inspected as formal quality control.
  10. Accepted summaries are combined for final scientific analysis.

Fundamental design rules

  • Never write output beside source videos.
  • Never write output beside master TOMLs.
  • Never modify a master TOML.
  • Every execution is a distinct immutable run.
  • Every run receives a unique run ID and complete settings snapshot.
  • IDtracker and post-processing are separate stages.
  • QC decisions refer to a specific run and a specific analysis unit.
  • Only accepted run-unit results belong in the final dataset.

Three-tab interface

1. Files & Runs

The app scans configured Firebird folders over SSH. It finds MP4, AVI, and TOML files, inspects TOML metadata, matches TOMLs to videos, and updates the local catalog.

A stable analysis-unit ID is generated from the video, TOML, and cell identity. A database primary key and unique path constraint prevent repeated scans from creating duplicate rows.

2. Settings

The user selects analysis units and creates a run. Each run includes:

  • run ID
  • run label
  • notes
  • execution mode
  • assay profile
  • minimum and maximum blob area
  • background-difference threshold
  • copied TOMLs
  • checksums
  • video and source TOML paths
Threshold edits are applied only to copied TOMLs inside the run folder. The application deliberately refuses to use the source TOML as the editable file.

3. Status & QC

This tab supports SLURM submission, status polling, job cancellation, result retrieval, track PNG display, and formal QC decisions.

Quality-control model

Each track is assigned to a specific run and analysis unit. The reviewer chooses:

  • accepted: suitable for final analysis
  • rejected: run should not be used
  • rerun needed: settings or tracking must be changed
  • review later: no final decision yet

The QC record stores reviewer, timestamp, notes, and track path.

Why accepted results are separated

Threshold testing and reruns are scientifically useful, but exploratory results must not silently enter the final dataset. The accepted-results table is therefore generated only from explicit QC decisions.

Installation

cd idtracker-pipeline-controller
bash scripts/install_mac.sh

Edit config/config.toml, then launch:

bash scripts/launch_app.sh

Required Mac software

  • Miniconda or another Conda distribution
  • system SSH client
  • rsync
  • a web browser

macOS normally includes SSH and rsync.

Required Firebird setup

  • working SSH access
  • permission to read video and TOML roots
  • permission to write to the pipeline run root
  • working IDtracker Conda environment
  • working post-processing environment
  • permission to submit SLURM jobs

Configuration fields that must be verified

  • SSH host alias
  • remote project root
  • video search roots
  • TOML search roots
  • IDtracker activation command
  • IDtracker execution command
  • post-processing activation command
  • post-processing execution command
  • GPU and CPU partitions
  • SLURM time, memory, CPU, GPU, and array limits

Expected future development

  1. Test exact IDtracker command on Firebird.
  2. Test exact post-processing command.
  3. Add automatic session discovery after IDtracker completion.
  4. Add dependency chaining between IDtracker and post-processing.
  5. Add per-array-task status parsing.
  6. Add thumbnail-to-analysis-unit matching.
  7. Add combined accepted-summary generation.
  8. Add multi-user roles and audit controls.
  9. Optionally move the browser app to Firebird or Open OnDemand.
The first release is an MVP. Do not submit hundreds of production jobs until one behavioral-assay cell, one fight cell, one multi-cell video, and one rerun with changed thresholds have been verified.

Minimum validation plan

  1. Scan Firebird twice and verify that row counts do not double.
  2. Create a one-cell behavioral-assay run.
  3. Verify that the source TOML remains unchanged.
  4. Verify that the copied TOML contains the requested thresholds.
  5. Submit IDtracker and inspect logs.
  6. Run post-processing and retrieve the track PNG.
  7. Record each QC decision type.
  8. Export accepted results.
  9. Repeat the same cell with different thresholds and confirm both runs remain visible.
  10. Repeat with a two-beetle fight cell and secondary ROI.
